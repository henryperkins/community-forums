§1 · Overview

The Community Inbox

One durable knowledge space, read like an inbox: the inbox is personal, the topic is durable, and the composer is immediate.

RetroBoards is self-hostable forum software that presents classic message-board depth through an interface people already use daily. The default authenticated home is not a forum homepage; it is a personalized forum inbox, and the durable unit is the topic. That sentence is the lens for every decision in this document.

Three lineages combine, and naming them is the fastest way to predict how any screen should behave:

Discourse bones
Durable topics, categories, tags, search, solved answers, moderation — canonical knowledge that outlives the session.
Slack polish
Immediacy, mentions, reactions, lightweight replies, a rich composer — conversational warmth at the point of writing.
Outlook discipline
Triage, split-pane reading, unread state, focused filters, keyboard-driven density — catch up without opening everything.
Status, stated plainly
Phase 5 Gate A is accepted and default-on (ADR 0018, 2026-07-09): signed packages, declarative themes, the capability resolver, passkeys, generic OIDC, invitations, secrets, read-only API tokens, and webhooks all ship on for any install without an explicit override — each independently reversible via features.<flag>=false. Gate B (server extensions, governance, service principals, verified links) stays reserved and dark. Thread Intelligence is implemented but pre-flip (§8). Where a surface is mock-only or dark, this document says so rather than implying it ships.

§2 · Information architecture

One spine, three registers

A category has many boards; a board has many threads; a thread has many posts. Every route hangs off a user and that spine — member surfaces, the operator consoles, and the machine endpoints.

The shell is three panes — a channel rail, the inbox, and the conversation — and every member surface is a route into that spine. A micro-router (src/Core/Router.php) matches {id} and {slug} patterns onto controller pairs; everything is server-rendered with progressive enhancement, and every flow has a no-JS fallback.

§3 · Screens and flows

Reading, posting, proving who you are

Each surface is given the same way: what it is for, the parts it is built from, and the flow that runs through it.

3.1 · Pane 3

The conversation

A thread opens as a chat-style message stream with the composer pinned to the bottom. The first post is the OP; consecutive posts by one author drop the repeated avatar and name.

Flow — reply
Type in the pinned composer → Enter sends → the write passes the gate below → redirect lands on the new post (/t/{id}-{slug}#p{postId}) with counters and unread markers updated in the same transaction. A reply held for review flashes “awaiting moderator approval” instead — the author is never dropped on an empty page.
3.2 · One component

The composer, every write

New topic, reply, DM, and edit share one composer: a canonical Markdown textarea, a Milkdown WYSIWYG layer over it, and a server-rendered preview through the same sanitise pipeline the post will use.

Drafts that survive
Local autosave always; signed in, drafts sync across devices (server_drafts) with a Drafts view and per-draft discard.
Media, held at arm's length
Images are sniffed, re-encoded, and stored outside the web root; delivery is authorization-gated per parent (/media/{id}).
Beyond text
One poll per thread with no-JS vote and results; operator-managed custom emoji shortcodes; a slash-command GIPHY picker, inert until a key is set.
The write gate
Every write runs the same sequence: named rate limits → anti-abuse rules (observe → flag → hold → block; automated scoring is capped at hold) → sanitisation → transactional counters. New accounts face stricter link and upload ceilings until they have a footprint.
§4 · Data model

What the screens are made of

InnoDB, utf8mb4, BIGINT surrogate keys, UTC datetimes, soft-deletes wherever history matters. Read-side counters are denormalised and maintained on write. 115 tables across 77 migrations — but the spine is still six.

-- a topic is the durable unit
threads(
  id, board_id, user_id,
  title, slug,
  is_pinned, is_locked, is_deleted,
  reply_count, view_count,
  last_post_id,       -- drives unread
  last_post_user_id,
  last_post_at, created_at )

-- a post is one message
posts(
  id, thread_id, user_id,
  parent_post_id,
  body, body_html,       -- raw + sanitised
  is_op, is_anonymous,   -- masked identity
  is_deleted, deleted_by,
  edited_at, edited_by )
Unread is derived
A thread is unread when thread.last_post_id > thread_user.last_read_post_id — no per-post read rows.
Sessions, not cookies-as-truth
Opaque tokens in a sessions table power the device list and log-out-everywhere; cookies are HttpOnly · Secure · SameSite. A guest is simply a request with no valid session.
History over deletion
Edits keep post_revisions; status changes keep thread_status_history; role grants, owner transfers, package installs, and appeals each keep their own ledger.

§5 · Status and authority

State you can read, power you can audit

Status is never carried by colour alone — every state pairs a hue with a word. Capability accumulates — Admin ⊇ Moderator ⊇ User ⊇ Guest — and since Phase 5 every check names one of 54 catalogued capabilities.

Topic status chips
Solved Needs answer Decision Pinned Locked Archived

The four fixed roles remain the spine, but enforcement now runs through an authority gate with three postures. Legacy (flag off): the fixed-role checks decide alone. Shadow — the shipped default (CAPABILITIES_MODE=shadow): legacy still decides, while the database-backed resolver dual-runs every gated read for mismatch telemetry. Enforce: the resolver decides, custom roles and scoped grants become live, and suspended staff lose their capabilities. The cutover evidence is 1,551/1,551 parity fixtures with zero mismatches.

Operators shape authority at /admin/roles — create or clone roles, grant per-capability, assign site- or board-scoped, with expiry, a full assignment history, and a permission simulator to answer “what could this person do?” before saving. Protected owners cannot be locked out; ownership moves only through audited transfer and recovery.

§8 · Thread intelligence

The Living Brief, pre-flip

AI proposes; the council approves. Evidence-bound briefs for eligible public threads, generated asynchronously, validated locally, published only when every claim traces to a source post.

The worker takes a bounded pass over queued public threads each minute, asks the provider for a structured brief, then runs it through local evidence, schema, content, and moderation checks before anything becomes visible. Curator edits remain authoritative input to later refreshes; a failed generation preserves the last good output; member-visible provenance links every claim back to its source posts. Personal since-last-read context stays deterministic — no model in that loop.

The boundary is deliberate: public post text only — no account, private, or moderation metadata leaves the install; raw prompts and responses are not retained; and with no credential configured, behavior is deterministic and non-AI. The credential lives only in the environment.
