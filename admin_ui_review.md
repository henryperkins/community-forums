# Admin Console UX Audit — findings report

**How this was audited:** I mapped all 100+ admin/mod routes in `src/Core/App.php:1621-1822` against ADMIN.md v0.13, then ran the app on a throwaway copy of the evidence DB (all Phase-5 dark flags enabled, seeded via `tests/browser/prepare.sh`) and drove every admin page plus the `/mod` queues with scripted Playwright — exercising real tasks end to end: suspend/lift/warn/note/badge a user, create/rename/archive/delete boards and categories, tag create/merge, badge-rule create→backfill→revoke, email test-send/suppress/unsuppress, announcement publish/clear, API token mint (wrong + right password)/revoke, webhook register/test/pause/delete, theme preview, role simulator, flag-off probes, a moderator-account walkthrough, and mobile-width checks. Three parallel code audits (CSP/PE/consistency, validation-failure handling, accessibility) fed in; I spot-verified every load-bearing claim at the cited lines. The DB was re-seeded pristine afterwards and no repo files were touched (`docs/evidence` PNG modifications in git status predate this session).

**Headline:** the foundations are genuinely strong — zero CSP violations, complete CSRF coverage, every mutation is a real form that works without JS, structure deletes have typed-confirmation with impact stats, and most forms re-render 422 with input preserved. The material problems are: **one live 500 on a routine daily action (suspend)**, a **family of draft-loss regressions on the `/mod` side**, a **phantom bulk-actions UI posting to a route that doesn't exist**, **moderators having no navigation into their own queues**, and several spec-promised operator surfaces (flags, roles, audit log, email templates) that simply have no UI.

---

## What's solid (baseline the findings sit on)

- **Strict CSP passes everywhere** — no inline `<script>`/`<style>`/`style=`/`on*=` in any of the 27 admin templates; the branding live preview uses CSSOM property assignment (`app.js:145-147`), which is CSP-legal.
- **Progressive enhancement is real**: every mutating action is a plain `<form method="post">` with `$this->csrfField()`; no GET mutates; reordering is per-row ↑/↓ button forms (keyboard-accessible); confirm flows are server-rendered GET pages. Zero console errors across the full walk.
- `structure_confirm.php:33-42` is the strongest destructive pattern: impact stats (`dl` of threads/posts), typed confirmation **enforced server-side** (`AdminController.php:124,254,339,362`), Cancel link, danger styling only for deletes.
- `/admin/users` is the best list page: labeled filters, `aria-sort` headers that preserve filters, pagination, search that covers email (`UserRepository.php:514`).
- Password re-auth consistently guards high-risk actions (tokens, webhooks, roles, theme activate, registry keys); registries even label their fail-safe asymmetry in the button text ("Disable registry (no password)" vs password-gated enable).
- Mobile (390px): no horizontal overflow on users/structure/email.
- Flag-off gated pages correctly 404 (never 403), and dark-flag nav items render as disabled spans with an explanation.

## Findings by page

### `/admin` — Dashboard (`templates/admin/dashboard.php`)

1. **[major] The console has no feature-flag control anywhere.** Nav items say "Disabled until the feature flag is enabled" (`_nav.php:5,37-40`) and `extensions.php:103-106` *describes* the emergency `server_extensions` flag — but no admin page can flip any flag; the operator's primary lever is hand-edited JSON in the `settings` table. Runbooks instruct "set `features.package_themes=false`" with no UI to do it. *Fix: a Settings → Features page writing the `features` override (or at minimum, document the SQL in the disabled-nav note).*
2. **[major] Overridden-off default-ON flags leave live links that 404.** With `email:false` in the features override, `/admin/email` 404s while the nav Email link (`_nav.php:7-16` renders the `$always` block unconditionally) and the "Email failures" dashboard card stay live — verified live. Dark flags get disabled spans; overridable ON flags don't. *Fix: run the `$always` items through the same flag check as the `$dark` list.*
3. **[minor] Settings surface is a fraction of ADMIN.md §9.2.** Spec promises Settings sections (General/Registration/Security/Email/Privacy/Legal/Advanced) with registration mode open/approval/invite, verification requirement, password policy, rate limits. Delivered: site name, registration **Open/Closed only**, anti-abuse mode, blocked words. Invite mode is a tracked Phase 5 deferral (`invitations` dark); approval mode and the rest are not tracked anywhere. *Fix: record the gap in an ADR or build the Registration/Security slice.*
4. **[minor] "New users today" (§9.2 dashboard spec) missing from the cards;** the Audit card links to the same page's fragment (`/admin#recent-activity`) whose feed is a fixed, unfilterable, unpaginated table — see the audit-log finding below.
5. **[polish] The member onboarding tour dialog ("Welcome… 1 of 5") floats over the admin console** (screenshot-verified, overlapping the Audit card) until skipped. *Fix: suppress `tour.js` on `/admin*`.*

### `/admin/users` — Directory (`templates/admin/users.php`)

6. **[major] The entire bulk-actions UI is a phantom.** The form posts to `/admin/users/bulk` (`users.php:94`) — **no such route exists** (`App.php` user routes end at `/lift`, line 1791; grep confirms zero matches); the select-all checkbox's `data-bulk-toggle` (`users.php:100`) has **no handler in any shipped JS**; the fieldset is permanently `disabled` (`users.php:135`) with "coming soon" (`users.php:146`), while the per-row checkboxes sit outside it and remain clickable no-ops. ADMIN.md §5.1 promises bulk-selectable; the repo's own rule says deferrals get an ADR — this one is in no plan or ADR. *Fix: remove the checkbox column + fieldset until the feature exists (and register the route when it does); record the deferral.*
7. **[minor] Global role isn't manageable and board-mod status isn't visible.** Alice (moderator of #general) shows plain "user" in the Role column with no indicator, and there's no way to change anyone's role (see finding 9). *Fix: add a scoped-mod badge sourced from `board_moderators`.*
8. **[minor/a11y] The one wide table without a focusable scroll region:** `users.php:96` `.table-scroll` lacks the `tabindex="0" role="region" aria-label` used by the 13 other instances (e.g. `dashboard.php:92`); sort arrows are literal `▲`/`▼` inside link text (`users.php:18`), announced by screen readers despite `aria-sort`.

### `/admin/users/{id}` — User record (`templates/admin/user_record.php`)

9. **[blocker] Suspend with a malformed "Until" value returns HTTP 500 and loses the typed reason.** The field is free text (`placeholder="YYYY-MM-DD HH:MM:SS"`, `user_record.php:73-74`) passed unvalidated into `bans.expires_at`/`users.suspended_until` (`UserModerationService.php:67-83`); under `ERRMODE_EXCEPTION` a value like `2026-13-99` throws PDOException → kernel 500. **Live-reproduced** ("Error 500", POST 500); the transaction rolled back cleanly (bob stayed active), so it's a validation/UX defect, not corruption — but suspending is a routine daily action performed while typing a justification. *Fix: validate/parse `until` into the existing ValidationException path (and consider `<input type="datetime-local">` + duration presets).*
10. **[major] No role assignment at all.** ADMIN.md §5.2/§5.3 promise assigning User/Moderator/Admin (with confirmation + audit for Admin grants). The record has title/badges/warn/note/suspend/ban only — granting Admin or site-wide Moderator requires SQL. Board-scoped moderators are assignable via the board edit page, which softens this, but admin succession has no UI path. *Partially superseded: the Phase 5 capabilities workstream owns role assignments (`PHASE_5_PLAN.md:445`, enforcement cutover Inc 6) — flag as tracked-but-distant; the legacy `users.role` gap is real today.*
11. **[minor] Spec'd record content missing:** email/verification status (PII-gated + audited per §5.2/§5.5), join date/last seen (shown in the directory but not here), posts/reports history, and IP/device signals (§5.5) are all absent. Ban form has no type (post-only vs full), scope, or duration (§3.4/§5.4) — "Permanent ban" only.
12. **[minor] Badge grant/revoke failures drop the typed reason** — `AdminUserController.php:117` re-renders 422 without old values (BadgeService VEs carry none), unlike warn/note/suspend on the same page which preserve input. *Fix: pass `['reason' => …]` like `warn` does at `:160`.*
13. **[minor] Posture inversion: banning a person is one click while deleting an empty board needs typed confirmation.** Suspend/Ban are single POSTs (`user_record.php:77,88`); reason-required is the only brake. *Fix: a confirm step (or type-the-username) for Ban.*

### `/admin/structure`, board edit + confirms (`structure.php`, `board_edit.php`, `structure_confirm.php`)

14. **[major] `post_min_role` is enforced but unconfigurable.** `BoardPolicy`/`PostingService` gate posting on it (`PostingService.php:208`), the setup wizard seeds `'user'` (`SetupService.php:116`) — and no admin template or `AdminController` handler mentions it (grep: zero). The ADMIN.md §4.2 headline use case ("Admin-only announcements" board) can't be built from the UI. *Fix: add a "Who can post" select to board create/edit.*
15. **[major] Deleting a non-empty board is a dead end.** The confirm page says "Move or delete its content before deleting the board" with no button — there is no bulk thread-move tool, so the operator must move threads one by one via each thread's mod control. ADMIN.md §4.4 promises a choice at delete time (move threads to another board, or delete with them). *Fix: add a "move all threads to …" select on the confirm page.*
16. **[minor] Board settings missing vs §4.2:** icon/emoji, edit window, locked state, allowed prefixes have no fields (create form is even thinner than edit — no approval/anonymous toggles until after creation). No bulk archive (§4.5). Categories: no default-collapsed flag (§4.1).
17. **[minor] Stale "Private (admins only)" label.** The visibility option (`board_edit.php`, also the create form) still describes the Phase-1 hold state, directly contradicting the Members section on the same page ("Members can read and post here when this board is private or hidden"). *Fix: relabel "Private (members only)".*
18. **[minor] Dead reorder scaffolding:** `data-reorder-categories`/`data-reorder-boards` (`structure.php:20,48`) have no JS anywhere in `public/assets/`, and their target `POST /admin/structure/reorder` (`App.php:1767`, handler `AdminController.php:294-320`) is unreachable from any UI. The ↑/↓ buttons are the real (good) mechanism. *Fix: delete the attributes + route, or ship the drag enhancement.*
19. **[minor/a11y] The category rename input has no accessible name at all** (`structure.php:27` — no label/aria-label/placeholder; a screen reader hears "edit text"), and since the category name exists only as that input's value, categories can't be enumerated by heading navigation. Add-category is placeholder-only (`structure.php:90`).
20. **[polish] "1 threads"** — board rows don't pluralize (`# Announcements … · 1 threads`).

### `/admin/tags` (`tags.php`, `TagController`)

21. **[major] Create and update lose everything typed on failure.** `TagController.php:98-99` (create) and `:120-123` (update) `redirectWithFlash('/admin/tags', $e->first())` on ValidationException — and the duplicate-slug `PDOException` path does the same — so name/slug/description (a 255-char field) vanish; update failures snap the row back to stored values. This is the clearest violation of the repo's own anti-draft-loss pattern on the admin side. *Fix: re-render 422 with `$e->old` like `AdminController::createBoard`.*
22. **[minor] Merge is irreversible mass-retagging presented as a muted one-click button** (`tags.php:53`, `linkbtn muted`, no confirm) — the weakest destructive posture in the console relative to consequence. *Fix: danger styling + a confirm step showing affected thread count.*
23. **[minor/a11y] Every per-row edit control is unlabeled** (`tags.php:32-38`: three inputs + a select with no label of any kind) — ironically the merge select two lines down does it right with a per-row sr-only label (`tags.php:45-46`).

### `/admin/email` (`email.php`, `AdminEmailController`)

24. **[minor] Contradictory status copy:** "Sending is configured. The delivery worker drains queued mail." immediately above "Set a From address before verifying SPF and DKIM" — configured-but-From-less reads as both fine and broken. (Domain send-blocking being opt-in is ADR 0008 by design — not a bug.) *Fix: one status block with explicit states: driver ✓ / From ✗ / domain unverified.*
25. **[minor] Delivery log has filters but no pagination** ("N total matching deliveries", `email.php:129`) — unlike the users directory; a busy install's log becomes filter-or-nothing. Suppress/unsuppress flows work well (live-verified with flashes); suppress input is placeholder-only, no label (`email.php:136`); "Requeue" flashes "Failed delivery requeued." even when the idempotent service did nothing (`EmailOpsService.php:133-141`).
26. **[minor] Spec gap (untracked): no email template editing.** §7.4/§7.5 promise editable templates with preview + test-send (Phase 2 per §11); only the fixed test-send exists. Similarly the §7.2/§7.4 staff-alert matrix (event × channel, thresholds, quiet hours) has no UI. Neither appears in any plan or ADR.

### `/admin/announcements` (`announcements.php`)

27. **[minor] The `announce` rate limit fires before validation and wipes the message.** `AdminAnnouncementController.php:61` enforces the limiter ahead of the ValidationException-protected path; a 429 renders the kernel error page and the typed banner text is gone. *Fix: enforce the limit only on otherwise-valid submissions, or re-render the form on 429.* (Publish → banner-with-dismiss → clear all worked cleanly live; validation failures preserve input.)
28. **[polish] No history or preview** — past broadcasts (which §7.4 says are audited) aren't listed; you see only the current banner.

### `/admin/api-tokens` (`api_tokens.php`)

29. **[minor] On a wrong password, the typed name survives but chosen scopes and expiry reset** — live-verified (`nameKept: "Audit token", scopeKept: false`); the service VE carries no old values (`ApiTokenService.php:52-71`). For a form that intentionally demands your password (easy to fumble), the scope selection should survive.
30. **[minor] Tokens table has no empty state** (`api_tokens.php:54-75` — bare header row on fresh installs) while its sibling webhooks table says "No endpoints yet." Also: mint deliberately breaks PRG to show the one-time token (documented accepted wart, `AdminApiTokenController.php:52-56`) — worth a "don't refresh this page" hint since reload re-mints.

### `/admin/webhooks` + detail (`webhooks.php`, `webhook_detail.php`)

31. **[major] `POST /admin/webhooks/{id}/test` with a stale id is an unhandled 500.** `AdminWebhookController::test` (`:165-173`) has no try/catch and `WebhookService::sendTestEvent` throws `ValidationException(['webhook' => 'Unknown webhook.'])` (`WebhookService.php:195`) — the kernel doesn't handle VE. Reachable by testing from a second tab after deleting the endpoint. Every sibling handler in the controller catches VE. *Fix: catch → flash "Unknown webhook." → redirect to the index.*
32. **[minor] Delete is one unconfirmed click** (`webhook_detail.php:58-61`) with no password — while *rotate secret* on the same page requires one; deleting the endpoint and its delivery history is the more destructive act. Live-verified: single click → "Webhook deleted." *Fix: match rotate's password gate or add a confirm.*
33. **[minor] Pause/Resume flashes the generic "Webhook updated."** (`AdminWebhookController.php:136`) — live-verified; you can't tell from feedback whether you paused or resumed. Test-send feedback is "Test event queued. Run the webhook worker to deliver it." — honest, but the operator gets no delivery result until cron fires; a delivery-log pointer or auto-refresh hint would close the loop.

### `/admin/packages`, `/admin/registries`, `/admin/themes`, `/admin/roles`, `/admin/extensions` — Phase 5 deploy-dark surfaces (report UX within an in-progress workstream)

34. **[major, within dark surface] The entire package lifecycle gives zero success feedback.** All twelve mutating handlers in `AdminPackageLifecycleController` end in bare `$this->redirect(...)` — `redirectWithFlash` appears **0 times** in the file — so Enable/Disable/Pin/Update/Rollback/Uninstall land silently back on the page. Every other admin controller flashes. *Fix: add flashes; the layout already renders them.*
35. **[minor] Uninstall is a bare unstyled `<button type="submit">Uninstall</button>`** (`package_detail.php:210-213`) — no `.btn`, no danger class, no confirm (password only); in fact all ~11 lifecycle buttons on the page are unstyled, an island of undesigned UI. `package_plan.php:75`'s button label *says* "(disabled until consent)" but the control is enabled and submits. `package_consent.php` is a dead end with no back/cancel link.
36. **[minor] Registries: the per-key revoke form's reason and password inputs are placeholder-only** (`registries.php:39-40` — a placeholder-only *password* field never gets an accessible name); blocklist table and zero-registries case lack empty states.
37. **[polish] Roles: the 48-capability checkbox list is one flat run** (`roles.php:52+`) — grouping by the self/board/site tiers it already implies would help scanning. Simulator + clone work; clone failure is the one flash-based error in the file, losing the typed name (`AdminRoleController.php:153-154`) vs create/update's 422+old.
38. **[polish] Extensions: the hand-maintained activation matrix** (`extensions.php:11-47`) hard-codes per-flag statuses that will drift from `FeatureFlags` reality. Theme safe-mode page correctly stays reachable with the flag off (deliberate recovery posture) — good.

### `/mod/reports`, `/mod/approvals`, `/mod/appeals` — the moderator surface

39. **[major] Moderators have no route into their own queues.** Logged in as alice (moderator of #general): the topbar/sidebar contain **zero** links to `/mod/*` (live-verified: no staff links on any page), `/admin` is a bare 403 error page, yet `/mod/reports` serves her a working scoped queue. ADMIN.md §9.1-9.2 promises a "reduced Console" for moderators. The queues exist; they're just undiscoverable. *Fix: show a "Moderation" topbar link (with open-count badge) to users with any `board_moderators` row; give the 403 page a pointer to `/mod/reports`.*
40. **[major] Approvals flash success even when the action failed.** `ApprovalController.php:76-80,95-99` flashes "Topic approved and published." regardless of `approvePendingThread()` returning false (e.g. lost race with another mod) — misleading feedback in the one queue where two people plausibly act at once. Also: reports is gated by `moderation_queue` while approvals has **no flag gate** — inconsistent.
41. **[major] The `/mod/u/*` moderation actions lose the typed reason on any failure.** `UserModerationController.php:71-73` catches ValidationException and redirects to the profile with only `$e->first()` — a moderator's multi-sentence warn/suspend/ban justification is gone; the *identical* service actions on `/admin/users/{id}` re-render 422 with input preserved. Same pattern loses the split-thread title + reply selection (`ModerationController.php:71-75`). *Fix: render the profile/thread view at 422 with old input, or carry old values through the redirect.*
42. **[minor] Reports queue is far short of §3.2:** no filters (board/reason/status/age), no sort, no bulk actions, no SLA aging cues, no author-history context on items — currently a plain list with claim/resolve/dismiss. Untracked as a deferral. **[polish]** All three queue templates render two `<h1>`s (`mod/reports.php:11,24`, `approvals.php:13,26`, `appeals.php:8,21`).

### Site-wide admin cross-cutting

43. **[major] No audit-log screen.** §3.6/§9.2 promise a filterable audit view (Admins site-wide, mods per-board via `mod.log.view`). What exists: the dashboard's short "Recent activity" feed (no filters, no pagination, no search) and per-user trails on records. Incident investigation ("who deleted what last Tuesday") means SQL against `moderation_log`. Untracked in any plan/ADR. *Fix: an `/admin/audit` page reusing the users-directory filter/pager pattern.*
44. **[minor/a11y, systemic] Field errors are never programmatically associated** — the house pattern is a sibling `<p class="field-error">` with no `id`/`aria-describedby`/`aria-invalid` and no error summary or focus management on 422 (instances across ~12 templates; `board_edit.php:22,27` shows the correct nested pattern). Data tables have no `<th scope>` or captions anywhere; action columns are headerless empty `<th>`s; repeated per-row buttons ("Revoke", "Save", "Merge") are undifferentiated to screen readers except in `board_edit.php:87,114`, which shows the right aria-label pattern.
45. **[minor] Destructive-action grammar is inconsistent** (full inventory verified): typed-confirm for board/category delete *and* archive/unarchive (archive being reversible makes this the strictest flow in the console) — vs one-click for ban/suspend, webhook delete, token revoke, badge-award revoke; **branding "Reset to defaults" is destructive but styled `btn-secondary` with no confirm** (`branding.php:115-119`); badge revoke and tag merge are styled *muted*. *Fix: one rule — reversible = single click + danger styling, irreversible = confirm step.*
46. **[minor] Silent failure under a success flash:** a corrupt logo/favicon upload is swallowed (`BrandingController::storeAsset` catches ValidationException and returns, `:211-215`) and the page still flashes "Branding updated." with the old asset in place.
47. **[polish] Terminology drift:** Remove vs Delete vs Revoke for like actions; `active/paused` vs `Enabled/Disabled` vs `active/revoked` state pills with varying capitalization; eight different Save-button labels; "Admin mode" pill everywhere except safe-mode's "Recovery".
48. **[minor] Orphan admin endpoints:** custom-emoji (3 POST routes, `App.php:1743-1745`) and link-previews (2 routes, `:1741-1742`) have **no rendering UI anywhere** — flag-dark today (tracked), but the emoji success flash even redirects to `/admin`, a page with no emoji section; if the flags graduate as-is, they're curl-only features. Flash is also single-slot (`Flash::add` overwrites), so multi-effect actions can only report one message.

## Known deferrals — checked and *not* reported as bugs

Verified against `docs/adr/0002-0012`, `PHASE_4/5_STATUS.md`, `docs/evidence/deploy-dark-features.md`: the 26 deploy-dark flags (appeals, custom_css, capabilities, package_registry/themes, api_tokens, webhooks, server_extensions, custom_emoji, link_previews, split_merge, invitations, governance, …) are intentional; packages "install-first" posture and roles "inert until resolver parity" are stated in-UI; email domain send-blocking is opt-in per ADR 0008; badge backfill being manual-only is recorded (`PHASE_4_STATUS.md:124`); the four PR-#33 theme fail-dark follow-ups are known review items. ADMIN.md §11's "Phase 1 not yet built" text is stale spec prose, not a live gap. **Conversely, four spec promises appear in *no* plan or ADR and are silent deferrals by the repo's own definition: bulk moderation (shipped as dead UI), the audit-log screen, email-template editing, and the staff notification matrix.**

## Top 10 by impact on a real operator's day

1. **Suspend "Until" → HTTP 500 with reason lost** (finding 9) — a routine trust-and-safety action, live-reproduced; fix is a date validation.
2. **Moderators can't find their queues** (39) — every moderator, every session, needs a memorized URL; `/admin` 403 is a dead end.
3. **Draft-loss family on moderation actions** (41, 21, 12, 37) — `/mod/u/*` reasons, tag create/edit, role clone, badge reasons; long-typed justifications are exactly what must never be dropped.
4. **No feature-flag UI + live nav links that 404 when flags are overridden off** (1, 2) — the console references flags it can't manage.
5. **No audit-log screen** (43) — accountability data exists, incident work requires SQL.
6. **Phantom bulk-actions UI posting to a nonexistent route** (6) — dead checkboxes today, guaranteed 404 the day the fieldset is enabled; untracked.
7. **No role management** (10) — admin succession / site-mod grants are SQL-only (Phase 5 will supersede, but that's the enforcement-cutover away).
8. **Webhook test 500 on stale id + silent package lifecycle** (31, 34) — the two "operator acts, system misleads" defects on integration surfaces.
9. **Approvals flash success on failed actions** (40) — wrong feedback in the queue where races are routine.
10. **Board management gaps: `post_min_role` unconfigurable + non-empty board delete dead end** (14, 15) — blocks the announcements-board use case and makes board retirement manual drudgery.

Everything above cites the exact route/template/controller lines; if you want, next steps could be turning the top items into a fix plan (the suspend-500, webhook-test-500, and the draft-loss redirects are each small, testable changes) or an ADR recording the four untracked deferrals.
